# TradingBot

An algorithmic **day-trading bot for US30 (Dow Jones CFD)**, built around an
**Opening Range Breakout (ORB)** strategy with a **machine-learning trade filter**,
running live against **MetaTrader 5**.

The trading rule is deliberately simple. The interesting part — and the bulk of
the work — is the **ML pipeline that decides which breakouts are worth taking**,
and the discipline used to make sure that pipeline isn't quietly fooling itself.
This page is written to show the ML engineering, not just the trading idea.

## The problem, framed as ML

A raw ORB breakout on US30 is barely profitable — backtested honestly with a
realistic ECN spread, it sits around break-even with a ~35% win rate at 2R. The
edge it *does* have is buried in noise. So the task isn't "predict the market";
it's a **binary classification problem**:

> given everything observable **at the moment a breakout fires**, estimate the
> probability that this specific trade reaches +2R before −1R, and only take the
> trades whose calibrated probability clears a learned threshold.

Framing it this way makes the whole thing measurable: the model is judged on
**calibration and expected R per trade**, not on accuracy, and the deployment
decision is driven by **out-of-sample expectancy**, not by a pretty backtest.

## Data and feature engineering

The training set is **~3,000 historical ORB trades (2016–2025)**, generated by
replaying the strategy bar-by-bar over 8 years of M5 data and labelling each trade
by its real outcome (win = hit 2R, loss = hit stop). Every trade is described by
**67 features**, all computed **causally** from information available at entry:

- **Setup geometry** — opening-range size, breakout distance, where the
  volume-profile POC sits inside the range, candle/volume characteristics of the
  breakout bar.
- **Multi-timeframe context** — ATR, ATR ratio, RSI, Bollinger-band width, EMA
  slope and distance-from-EMA, and a trend flag, each computed on **M5, M15, M30,
  H1, H4 and D1** and joined to the trade by an as-of merge on the last *closed*
  bar of each timeframe.
- **Session / calendar** — hour, minute, day-of-week, month, and trade sequence
  within the day.

Features that looked predictive but were either non-stationary or carried no real
information (raw price levels, CFD tick "volume") were **dropped after analysis**,
not kept for decoration.

## Avoiding the ways this silently breaks

Most "ML for trading" projects fail not because the model is weak but because the
evaluation leaks. The pipeline is built around closing those leaks:

- **Time-ordered splits, never random.** Train / calibration / test are
  contiguous in time (70 / 15 / 15), so the model is always tested on the future
  relative to its training data.
- **Purged cross-validation with a gap and an embargo.** Inner CV uses
  `TimeSeriesSplit` with a gap between train and validation folds; the
  train/calib/test boundaries carry an **embargo** of several trades so that a
  trade whose D1 feature lookback overlaps the next segment can't leak its label
  across the boundary.
- **Threshold chosen on calibration, reported on test.** The operating threshold
  is selected on the *calibration* set (out-of-sample) and only then measured on
  the *test* set. Picking the threshold on the same data you report is one of the
  most common ways backtests get inflated — so it's explicitly avoided.
- **Walk-forward across 5 expanding windows.** The single split is a starting
  point; the trusted verdict comes from retraining over five expanding windows and
  checking the edge survives in each unseen slice.
- **A feature-parity test as the deploy gate.** A `pytest` golden-vector test
  asserts the **live** feature builder produces the *same* vectors as the
  **training** pipeline, to float tolerance, for a spread of historical
  timestamps. This caught a real look-ahead bug during development where the live
  builder and the training builder disagreed — exactly the kind of silent failure
  that makes a model trade on inputs it was never trained on.

## Models, calibration and selection

- **Ensemble of gradient-boosted trees** — XGBoost and LightGBM, each tuned by
  **Optuna** (200 trials, GPU) **optimizing the Brier score** (calibrated
  probability quality) rather than accuracy, with class imbalance handled via
  scale/weighting and early stopping on the calibration fold.
- **Probability calibration** — raw model scores are passed through **isotonic
  regression** fit on the held-out calibration set, so a "0.6" genuinely means
  ~60% win rate. Calibration is verified with **reliability diagrams** and Brier /
  log-loss, because the whole strategy depends on the probabilities being honest,
  not just well-ranked.
- **Interpretability + parsimony** — **SHAP** values rank feature contribution,
  and the model is retrained on the top ~70% of features to check the edge isn't
  resting on a handful of fragile inputs.
- **Risk-of-ruin modelling** — bootstrap confidence intervals on win rate / avg-R
  / Sharpe, plus a **Monte-Carlo trade-shuffle** to estimate the max-drawdown
  distribution and the probability of breaching a prop-firm drawdown limit at a
  given risk-per-trade.

## The honest question: does the ML actually help?

A filter that just thins out trades can look good by luck. So the pipeline runs an
explicit **ML-vs-baseline test**: it compares the avg-R of the ML-selected trades
against simply taking *every* rule-filtered signal, and **bootstraps the
difference** (5,000 resamples) to get a confidence interval on the lift. The model
ships only if the lower bound of that interval stays above zero — i.e. the edge is
statistically real, not a small-sample mirage.

That same test is what **killed a 1:1 high-win-rate variant**: it produced a
prettier ~63% win rate, but the bootstrap CI for its expectancy **included zero**
and one walk-forward fold went negative, so it was rejected in favour of the 2R
configuration whose edge held up across every fold.

> ⚠️ Backtest and walk-forward results do **not** guarantee future performance.
> All numbers above are **simulation** (with ECN spread modeled; slippage and
> open-spike spread are not). The bot has **never traded real money** — it is in a
> demo forward-test.

## Live execution & risk

1. At the New York open the bot builds the 15-minute range, takes the first M5
   close beyond it (long above / short below), filters weak setups by rule (range
   too small, weakest weekday, higher-timeframe RSI disagreeing), then scores the
   survivor with the calibrated ensemble and trades only above threshold.
2. Stop loss at the volume-profile **POC** of the range, take profit at **2R**;
   one re-entry allowed after a stop; flat by end of session.
3. **Risk management:** fixed 1% risk per trade, **halved after two consecutive
   losses**, with a **daily-loss breaker** and a **max-drawdown breaker**
   (prop-firm-style). Breaker state is persisted, so it survives a restart
   mid-session.

## Tech stack

- **Python** — core engine, strategy state machine, live runner
- **XGBoost / LightGBM** + **scikit-learn** — the calibrated trade-filter ensemble
- **Optuna** — Bayesian hyperparameter search (Brier-optimized)
- **SHAP** — feature attribution and selection
- **pandas / NumPy / PyArrow** — data pipeline and feature engineering
- **MetaTrader 5** API — live market data and execution (retry logic, spread
  buffer, position sizing)
- **pydantic / pydantic-settings** — typed config and data models
- **CustomTkinter** — desktop dashboard (multi-account management, live equity /
  drawdown / breaker state, presets, logs)
- **pytest** — feature-parity (deploy gate) and regression tests
- packaged as an installable Python project (`src/` layout, `pyproject.toml`,
  console entry points)

## Architecture (high level)

```
brokers/      MetaTrader 5 wrapper (orders, sizing, spread handling, retries)
strategies/   ORB state machine + live feature builder + model bundle loader
ml/           dataset builder; training in a Colab notebook (Optuna/CV/calibration/SHAP)
backtest/     backtest engine with SL/TP/filter grids
execution/    live runner (paper / demo / live modes)
utils/        risk manager, indicators, news filter, logging, time utils
gui/          CustomTkinter dashboard (multi-account, presets, live monitoring)
tests/        feature-parity golden-vector test (training == live)
```
